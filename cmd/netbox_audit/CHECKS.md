# Audit Checks

This page documents what each audit check verifies, the kind of mistake it is intended to catch, and the current boundaries of the check.

- `Required Device Fields` (`required-device-fields`): Verifies that every device has the minimum inventory context the rest of the model assumes: site, role, and status. Example misconfiguration: a switch imported from another system exists in NetBox but has no role or site, so later audits cannot reason about it correctly. Current scope/exemptions: this is intentionally narrow; it does not require tenancy, serials, asset tags, manufacturers, or other optional inventory metadata. The reasoning is that this audit only enforces fields that the rest of this repo actively depends on.
- `Device Locations` (`device-locations`): Ensures every device has a physical location when location is expected to be part of the model. Example misconfiguration: a compute node exists in inventory but has not been assigned to `Garage`, `Network Closet`, or another defined location. Current scope/exemptions: there are no built-in exemptions right now, even for intentionally unplaced devices. The reasoning is that these omissions are still worth surfacing as planning debt.
- `Parent Placement Consistency` (`parent-placement`): Checks that child devices agree with their parent device on shared placement fields such as site, rack, and location. Example misconfiguration: a shelf-mounted amp is modeled as a child of a rack shelf, but its rack or location points somewhere else. Current scope/exemptions: missing `location` or `site` is expected to be caught by the dedicated completeness checks, so this check only compares those fields when both sides have values. Rack is treated more strictly: if either side has a rack and the other does not, that is flagged as inconsistent. The reasoning is that this check is meant to catch contradictory placement claims, and a parent/child rack mismatch includes the case where only one side claims rack placement.
- `Rack Placement` (`rack-placement`): Verifies that rack-mounted devices which are not explicitly exempt have a rack position and face. Example misconfiguration: a switch is assigned to a rack but has no U position, or has a position but no front/rear face. Current scope/exemptions: child devices are exempt when `rules.rack_placement.exempt_child_devices` is true, and tagged exemptions such as `0u-rack-device` are also skipped. The reasoning is that shelves and vertical PDUs do not fit NetBox’s normal U-position model cleanly.
- `Device Type Drift` (`device-type-drift`): Compares device instances to the components expected from their device type and installed modules. It catches missing, extra, or mismatched interfaces, ports, bays, and module-derived components. Example misconfiguration: a patch panel instance has the wrong rear ports for its installed keystones, or a device interface type was manually changed away from the device type definition. Current scope/exemptions: it only audits component classes loaded by this tool, and it treats installed modules as part of the expected shape. The reasoning is to catch meaningful structural drift without requiring every possible NetBox object type to be modeled.
- `Honeypot Coverage` (`honeypots`): Verifies that each VLAN-backed subnet has at least one IP tagged `honeypot`, and that each honeypot IP actually sits inside a VLAN-backed subnet. Example misconfiguration: the Power subnet exists in NetBox but no honeypot IP has been defined for it. Current scope/exemptions: it only considers prefixes that have an associated VLAN, so non-VLAN prefixes such as WireGuard client space are ignored. This matches Unifi's internal restrictions. Honeypots are created within a VLAN, and no more than one is allowed in any VLAN.
- `Wireless Normalization` (`wireless-normalization`): Checks that unsuppressed Wi-Fi interfaces on non-AP devices have the metadata needed to reason about them: wireless mode, untagged VLAN, and primary MAC address. Example misconfiguration: a Wi-Fi-only console has no access-mode VLAN, or a station interface has MACs recorded but no primary MAC selected. Current scope/exemptions: Access Points, planned devices, disabled Wi-Fi interfaces, and Wi-Fi interfaces on devices that already have a sufficiently modeled connected wired interface are skipped. The reasoning is to avoid false positives for fallback radios that do not matter operationally when wired connectivity is authoritative.
- `PoE Power Sufficiency` (`poe-power`): Ensures that interfaces modeled as powered devices (`pd`) are connected to peers modeled as power sourcing equipment (`pse`) with sufficient PoE type. Example misconfiguration: a ViewPort requires PoE but the connected switch port is not marked as PSE, or is marked with a weaker PoE type than required. Current scope/exemptions: it only evaluates enabled interfaces marked `pd` with connected peer interfaces present in the snapshot. It does not automatically model injectors or infer PoE through missing intermediate devices; those appear as warnings unless `rules.poe` is relaxed. The reasoning is to prefer explicit power-path modeling over guesswork.
- `Interface VRF Coverage` (`interface-vrf`): Checks that in-use interfaces have a VRF unless they are intentionally WAN-side exceptions. Example misconfiguration: an active switch uplink or access port has connected endpoints and VLAN data but no VRF. Current scope/exemptions: planned devices, disabled interfaces, WAN-side interfaces, and effectively inert interfaces with no meaningful usage signals are skipped. The reasoning is that VRF coverage only matters for interfaces that are actually participating in the modeled network.
- `Private IP VRF Coverage` (`private-ip-vrf`): Enforces the policy that RFC1918 addresses should belong to a VRF, while public-IP behavior is configurable. Example misconfiguration: a `10.x.x.x` address exists in NetBox with no VRF assignment. Current scope/exemptions: this check operates only on the IP objects themselves; it does not care whether the IP is assigned to an interface, device, or VM. Public-IP enforcement is optional via config. The reasoning is to keep this rule simple and independent from interface modeling.
- `IP / VLAN Consistency` (`ip-vlan`): Confirms that an interface’s access VLAN agrees with the VLAN inferred from the best matching prefix for its assigned IP. Example misconfiguration: an interface is modeled as access on `Direct Clients`, but the assigned IP falls inside a `Development` subnet. Current scope/exemptions: it only evaluates IPs assigned to interfaces that are explicitly modeled as access ports with an untagged VLAN. Trunks, hybrid-like cases, and interfaces without IP assignments are skipped. The reasoning is that the access-port case is the one where a clear one-to-one VLAN expectation exists.
- `Cable Consistency` (`cables`): Verifies that every cable has a type, a status, and both A and B terminations. Example misconfiguration: a patch cable exists but one end is missing, or the cable type was never recorded. Current scope/exemptions: it does not attempt to judge whether the chosen cable type is physically appropriate for the linked terminations. The reasoning is that this is a completeness check, not an electrical/media validation engine.
- `Patch Panel Continuity` (`patch-panel`): Checks that front and rear patch-panel ports with cables are actually mapped across the panel. Example misconfiguration: a rear port is connected to a switch, but there is no front-side mapping showing which front jack that termination corresponds to. Current scope/exemptions: it only checks for the existence of mappings on cabled front/rear ports; it does not prove a full end-to-end path from client device to switch. The reasoning is to catch the most common half-modeled panel mistakes with a low false-positive rate.
- `Module Consistency` (`modules`): Validates that installed modules, module bays, and module-bay pointers agree with one another. Example misconfiguration: a module record points at a bay owned by a different device, or a bay says it contains one module while the module list says otherwise. Current scope/exemptions: it does not currently validate whether a given module type is allowed in that bay according to any higher-level design rule. The reasoning is that referential consistency is more critical than policy constraints for the current model.
- `MAC Consistency` (`macs`): Looks for duplicate MAC records and interfaces with multiple MACs but no primary MAC chosen. Example misconfiguration: the same MAC address appears on two different interface assignments, or a multi-MAC interface is ambiguous for DHCP export. Current scope/exemptions: it checks NetBox’s recorded MAC objects and interface MAC associations only; it does not compare them to live UniFi state. The reasoning is to keep the audit snapshot self-contained and deterministic.
- `DHCP Reservations` (`dhcp-reservations`): Ensures that `dhcp-reserved` IPs are assigned to interfaces, live inside a `dhcp-reserved` range, do not rely on ambiguous MAC data, and do not overlap DHCP pool ranges. Example misconfiguration: a reserved IP is tagged correctly but sits outside the reserved range, or the interface it is assigned to has no usable MAC. Current scope/exemptions: it keys off the `dhcp-reserved` tag rather than IP status, and it does not require a DNS name. The reasoning is that tags are the authoritative workflow signal in this model, while DNS is intentionally optional.
- `Planned Device Hygiene` (`planned-devices`): Ensures planned devices do not look partially deployed. Example misconfiguration: a device still marked `planned` already has cables attached, MAC data present, or IP addresses assigned. Current scope/exemptions: it only inspects interface-level evidence of deployment, not rack position, location, or textual notes. The reasoning is that interface state is the strongest signal that a planned device has started becoming real.
- `Switch Link Symmetry` (`switch-link-symmetry`): Compares switch-to-switch links on both ends to make sure their mode, native VLAN, and tagged VLAN sets agree. Example misconfiguration: one end of an uplink is an all-VLAN trunk while the other end is modeled as access or has a different native VLAN. Current scope/exemptions: it only checks cables that directly connect exactly one interface on one switch to exactly one interface on another switch. It does not reason through patch panels, port-channels, or non-switch intermediary devices. The reasoning is that direct interface-to-interface links are the most reliable place to enforce strict symmetry.
